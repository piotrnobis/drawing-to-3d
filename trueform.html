01 / the problem

A drawing is not a model.

Industry runs on millions of legacy 2D drawings with no 3D counterpart. To remake a part, an engineer rebuilds it by hand in CAD — slow, and exactly the structured visual reasoning a model should do. But two things make it hard.

It must be editable

The output has to be a parametric STEP B-rep, not a mesh. A triangle soup looks fine and is useless — you cannot change a wall thickness or re-fit a bore on it.

Looking right isn't enough

Inferring one solid consistent across three views is genuine spatial reasoning, and vision models are weak at it. A part that looks right but measures wrong is worthless. So checking is the product, not an afterthought.

02 / overview

One loop: understand → build → prove → fix.

Everything happens inside a single Gemini conversation, so each step carries the full context of the ones before it. The agent reasons about the drawing, writes CadQuery code, runs it in a sandbox, then verifies the result with two independent signals and refines until both pass.

03 / the pipeline, as built

Six steps, three Python packages.

The backend is real and runs end-to-end today. Each step maps to a module; the LLM layer, the sandboxed CAD runner, and the agent that orchestrates the loop.

Step 01 · read

Analyze

Gemini studies the drawing first — what the part is, a per-view breakdown, and a structured dimension table (every callout with value, tolerance and a typed kind). This table is also the ground truth the size check uses later.

backend/agent/analysis.py
Step 02 · build

Generate

Grounded by its own analysis and an embedded CadQuery manual (mental model, design steps, pitfalls), the model writes one parametric script — Z-up, real CAD operations, ending in a single solid.

backend/agent/prompts.py
Step 03 · run

Execute & repair

The untrusted script runs in a sandboxed subprocess (scrubbed env, time-limited). If it errors, the traceback goes back into the same conversation and the model fixes it — cheaply, before any verification.

backend/cad/render.py · _harness.py
Step 04 · observe

Render & measure

From the solid we export STEP/STL and render four views (front/top/side/iso). We also measure the B-rep: bounding box, hole patterns (count, pitch, bolt-circle), and a connectivity count.

backend/cad/_harness.py
Step 05 · prove

Verify (2 signals)

Shape: Gemini compares the renders to the drawing. Size: a numeric gate checks the measured geometry against the dimension table. A model is verified only if both pass.

backend/agent/agent.py · gate.py
Step 06 · fix

Refine (keep best)

If a check fails, the shape critique and the failing numbers are fed back and the model regenerates. Bounded iterations, and elitism — we always return the best candidate seen, never a regression.

backend/agent/agent.py
04 / the idea that wins

We don’t just generate. We verify — two ways.

The eye and the ruler catch different errors, so we use both, and we divide the labour cleanly: the vision model judges shape, the geometry kernel judges size. Neither is asked to do the other’s job.

Signal 1 · shape (the eye)

Does it look like the part?

Gemini sees the four rendered views beside the drawing and reports structural differences only: a missing or extra feature, a bore that should be open but is solid, a feature on the wrong face, or two parts that should be one but are disconnected. Exact sizes are explicitly left to the gate.

A deterministic connectivity check backs it up: if the B-rep is more than one solid, the part fell apart — caught instantly, no LLM needed.

Signal 2 · size (the ruler)

Does it measure right?

We query the actual solid and check each callout within tolerance. Beyond overall size and bore diameters, the gate now verifies hole patterns precisely — count, hole-to-hole pitch, and bolt-circle diameter — per flange. That catches mis-placed screw holes the eye cannot judge.

The gate has teeth. Inject a fault — build a 30 mm bolt pattern where the drawing calls 40 — and the gate fails it deterministically: “hole pitch: expected 40 ±0.5 mm but measured 30 mm”. The eye would wave that through; the ruler does not.

06 / engineering & guards

How it stays robust

  • CTXOne conversation, many roles. Analyze, generate, repair, critique and refine are turns in the same chat — the model fixes its own prior code with full context.
  • JSONStructured output. Reasoning + code come back as typed JSON, so extraction is robust; salvaged gracefully if a reply is ever truncated.
  • SECSandboxed execution. Model-written CadQuery never runs in our process — only in a subprocess with a scrubbed environment and a hard timeout.
  • NETResilient calls. Transient API errors retry with backoff; long multi-step runs survive a dropped connection.
  • DOCDomain manual as context. An embedded CadQuery guide (patterns + real pitfalls we hit) lifts first-try quality.

Stack

  • LLMGemini — one vision model, structured output, high media resolution.

07 / scope & next
Built today (backend)

The full agent loop: analyze, generate, sandboxed run + repair, multi-view render, B-rep measurement, two-signal verification (shape critique + dimension gate incl. per-feature holes), bounded refine with keep-best, versioned traces.

Next

A web UI/API wrapping the loop (live drawing → spec → 3D viewer → pass/fail table); project-back overlay for a second shape signal; measuring more dimension kinds (thickness, feature spacing); more hero part classes.

Honest scope. We win by nailing a few real part classes end-to-end with a passing dimension table — not by gesturing at every possible part. Today it’s a backend that demonstrably reconstructs and verifies a multi-flange manifold; the UI is the wrapper, not the substance.
